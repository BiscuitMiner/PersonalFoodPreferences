# PersonalFoodPreferences — 食物分類補全提示詞

## 使用方式

當使用者提供一個 RimWorld 模組路徑，例如：

```text
D:\Game\Steam\steamapps\workshop\content\294100\<WorkshopID>
```

請依照本文件分析該模組的食物、飲品與可攝取物，判斷是否需要為 PFP 新增或更新 `FoodOverrideMapDef`。如果使用者同時提供 `.json` 分類檔，優先讀取 JSON；如果沒有 JSON，則從模組 XML、RecipeDef、ThingDef、ThoughtDef、語言檔與描述自行推斷分類。

任務目標是讓第三方模組食物能被 PFP 正確歸入 13 個偏好類別，避免所有食物都只落入泛用 `Meal` 或 `GenericFood`。

## 專案背景

PersonalFoodPreferences 將食物分類為 13 個偏好類別：

`Meat` `VeganMeal` `Baked` `Sweets` `Soup` `Canned` `Fruit` `Seafood` `Dairy` `SoyProduct` `Barbecue` `Fried` `DarkCuisine`

所有食物分類資料應放在 XML Def 中。C# 僅作為分類引擎，不得為單一模組硬編碼 `defName`、字串關鍵字或數值。

## 分類優先級

PFP 分類順序如下：

1. `FoodCategoryExtension`：直接掛在 ThingDef 上的 ModExtension。
2. `FoodOverrideMapDef`：以 `defName` 精確匹配。命中後會跳過關鍵字匹配。
3. `FoodCategoryKeywordDef`：以 `defName` / `thingCategories` 關鍵字模糊匹配。
4. `FoodTypeFlags` 與食材分析：從原版食物旗標與實際食材推斷。

若第三方食物只有 `<foodType>Meal</foodType>`，通常需要精確 override，否則 PFP 難以分辨肉類、甜點、湯、烘焙、軍糧等細分類。

但不是所有 `Meal` 都應強行歸類。若料理本身只是「簡單料理、精緻料理、豪華料理、一般熟食」這類可由任意食材製作的通用餐點，且沒有固定純肉、全素、甜點、湯、罐頭等語意，允許在 override 中保留空的 `<primaryCategory></primaryCategory>`，讓 PFP 於執行期依實際食材分析分類。

## 是否納入分類

應納入：

- 有營養或明確作為食物的 `ThingDef`。
- 餐點、料理、湯、甜點、烘焙、罐頭、軍糧、乾糧。
- 生食材，如肉、魚、蔬果、乳品、豆製品，若模組新增且 PFP 不能自然判斷。
- 種族專屬食物，只要它是正常食物偏好的一部分。

通常不納入：

- 純藥物、針劑、血清、戰鬥強化劑。
- `drugCategory` 為 `Hard`、且描述/用途不是食物的物品。
- 只提供 hediff 或化學效果、沒有食物語意的飲品。
- 非人類可食用對象專用的飼料、機械燃料、特殊資源，除非 PFP 明確支援該用途。

有疑義時，先說明判斷理由，不要把藥物硬分類成食物。

## 分類準則

| 類別 | 使用時機 |
|---|---|
| `Meat` | 純肉料理；料理主要食材全部由肉類構成 |
| `VeganMeal` | 全素料理；不含肉、魚、蛋、奶、動物產品或屍體來源食材 |
| `Baked` | 麵包、派、披薩、餅乾、蛋糕胚、烘焙主食 |
| `Sweets` | 糖果、甜點、蜂蜜、巧克力、布丁、含糖飲品 |
| `Soup` | 湯、燉菜、粥、羹、broth、stew |
| `Canned` | 罐頭、軍糧、乾糧、醃製、煙燻、保存食、survival pack |
| `Fruit` | 水果、莓果、果汁類，若不是甜點飲料 |
| `Seafood` | 魚、壽司、海鮮料理 |
| `Dairy` | 牛奶、起司、優格、奶油、蛋類料理 |
| `SoyProduct` | 豆腐、豆奶、大豆加工品 |
| `Barbecue` | 烤肉、燒烤、grill、roast、kebab |
| `Fried` | 油炸、煎炸、deep fried、chips、cutlet |
| `DarkCuisine` | 人肉、昆蟲、扭曲肉、恐怖/異常料理、明顯禁忌食物 |

`primaryCategory` 放最能代表該食物的分類；`tags` 放次要特徵。例如：

- 烤牛排：`primaryCategory=Barbecue`，`tags=Meat`
- 純肉湯：`primaryCategory=Soup`，`tags=Meat`
- 蛋糕：`primaryCategory=Sweets`，`tags=Baked`
- 軍糧：`primaryCategory=Canned`，視食材加 `Meat` / `VeganMeal`
- 通用簡單料理/精緻料理：`primaryCategory` 留空，交給食材分析

`Meat` 與 `VeganMeal` 是互斥分類，不得同時加在同一個食物上。含蛋或奶的料理不算 `VeganMeal`；魚、貝與海鮮料理使用 `Seafood`，不使用 `Meat`。
通用 keyword 仍可能因第三方 `defName` 過於寬鬆而命中多個方向，最終由 `FoodClassificationNormalizer` 清理不可能共存的 `Meat` / `VeganMeal`；但新增 override 時仍應避免寫入衝突資料。

## XML 格式

建議大型或單一主題模組建立獨立檔案：

```text
Common/Defs/FoodOverrideMapDefs/<ModName>.xml
```

範例：

```xml
<?xml version="1.0" encoding="utf-8" ?>
<Defs>

  <!-- Example Mod (Steam Workshop 1234567890) -->
  <PersonalFoodPreferences.FoodOverrideMapDef>
    <defName>PFP_Override_ExampleMod</defName>
    <overrides>
      <li><defName>Example_GrilledSteak</defName><primaryCategory>Barbecue</primaryCategory><tags><li>Meat</li></tags></li>
      <li><defName>Example_BerryPie</defName><primaryCategory>Baked</primaryCategory><tags><li>Fruit</li><li>Sweets</li></tags></li>
      <li><defName>Example_CookedRice</defName><primaryCategory></primaryCategory></li>
    </overrides>
  </PersonalFoodPreferences.FoodOverrideMapDef>

</Defs>
```

規則：

- `defName` 必須唯一且語意化，建議 `PFP_Override_<ModName>`。
- `primaryCategory`、`fallbackCategory`、`tags` 必須使用合法 13 類；除非 `primaryCategory` 是刻意自訂顯示分類，此時必須提供合法 `fallbackCategory`。
- `primaryCategory` 可以刻意留空，用於無法由 ThingDef 固定歸類、必須看實際食材的料理。
- 不要新增無來源依據的分類。
- 不要在 C# 寫入模組專屬 `defName`。

## 執行流程

收到 mod 路徑後，依序執行：

1. 定位：讀 `About/About.xml`，列出 mod 名稱、`packageId`、版本資料、依賴。
2. 掃描：用 `rg "<ingestible>|<foodType>|tasteThought|RecipeDef|ThingDef"` 找食物與飲品。
3. 彙整：列出候選 `ThingDef.defName`、label、`foodType`、`drugCategory`、`nutrition`、`tasteThought`、配方食材。
4. 篩選：排除純藥物、針劑、血清、燃料、非食物資源。
5. 分類：依分類準則決定 `primaryCategory` 與 `tags`。
6. 查重：確認既有 override 沒有相同 `defName`。
7. 實作：新增或更新 XML。
8. 驗證：至少用 XML parser 驗證新增/修改檔案可解析。
9. 編譯：若只改 XML，可不立即編譯；若使用者要求批次處理，編譯放最後。

## 輸出格式

每次分析與實作後，回覆需包含：

1. **定位**：模組名稱、packageId、關鍵路徑、候選食物來源。
2. **方案**：是否新增 XML override；是否排除藥物/特殊物品；風險等級。
3. **實作 + 驗證**：列出新增或修改檔案、主要 defName、XML 驗證結果；若未編譯要明說。
4. **SoC 檢查**：確認資料在 XML、C# 無硬編碼、未修改原版或第三方模組檔案。

## PFP 缺陷回報

若分析中發現問題不是單一模組資料能解決，而是 PFP 程式邏輯缺陷，必須明說。例如：

- PFP 將藥物誤判為食物。
- PFP 未排除無食物需求的 pawn。
- PFP 無法從常見配方結構推斷食材。
- PFP 的關鍵字優先級造成明顯誤判。
- 正規化後仍出現 `Meat` / `VeganMeal` 同時命中。

此時先提出最小修正方案，再判斷是否需要 C#、XML 或兩者配合。
